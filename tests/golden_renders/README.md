# DuskAmp Golden Renders

This directory holds local WAV outputs from `tests/duskamp_golden_render/` — one file per factory preset, generated by running `DuskAmpProcessor` against a synthetic DI.

**The WAV files themselves are not committed** (covered by the repo-wide `*.wav` ignore). Each developer regenerates them locally as a baseline for the audio regression check, then re-blesses after intentional DSP changes.

## First-time setup

```bash
./tests/duskamp_regression_check.sh --build
```

The first run with no goldens present will exit early; render once with the renderer to bless your local baseline:

```bash
cmake --build build --target duskamp_golden_render -j8
./build/tests/duskamp_golden_render/duskamp_golden_render
```

That writes 30 WAVs (~330 MB total) here. Subsequent runs of `duskamp_regression_check.sh` md5-compare a fresh render against this set.

## After an intentional DSP change

If the regression check fails because you intentionally changed a DSP path:

1. Inspect the diff in the script output — it lists which presets changed.
2. AB-listen the affected presets (load the old WAV here against the rebuilt plugin's output).
3. If the change is desired, re-bless by re-rendering:
   ```bash
   ./build/tests/duskamp_golden_render/duskamp_golden_render
   ```
4. The next regression run will pass against the new baseline.

## Why not commit the WAVs?

330 MB of bit-identical float WAVs would balloon the repo and rot fast under any DSP iteration. The renderer is fast (~30 seconds for all 30 presets) and fully deterministic, so regenerating locally is cheaper than carrying the bytes in git.

If multi-developer regression coverage becomes important, switching to a committed `MANIFEST.md5` (3 KB of hashes) is a one-line change to the regression script.
